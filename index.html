Living with other people's code

  • i've worked on mostly agency work up until this point
  • i've worked on a few products but mostly green field stuff
  • this is the first time i've inherited a code base
  • I wanted the challenge of taking an existing product and improving it over time.
  • I regret everything
  • When I joined we had no module system, no way to trace dependencies. globals everywhere.
  • I had a crippling fear of breaking everything
  • I'm only now getting to grips with things
  • Here are some of the things I try to work and live by (with mixed success)
  • Some of it is common practice, but is worth mentioning anyway
  • Some are just general lifestyle suggestions

1. Start thinking of it as your code

  • There is no such thing as legacy code
  • Just code you dont like

2. Don't blame other people

  • Your predecessor will most likely have been presented with different priorities to yourself.
  • At an early stage, shipping is a high priority.
  • This is okay
  • It is impossible to know what new requirements will come along.
  • It is an awful, but fun challenge to improve things.

3. Change is inevitable

It is change, continuing change, inevitable change, that is the dominant factor in society today. No sensible decision can be made any longer without taking into account not only the world as it is, but the world as it will be. â€” Isaac Asimov

  • You cannot know with any degree of certainty what requirements will come up in the following months and years
  • Do your best with the limited information available to you.

4. Treat all code as technical debt

  • All code is a barrier to change
  • To change code you must first understand entirely what it does
  • Your job isn't done until you've removed all code from your application
  • Your job is never done

5. Delete code

  • If no one uses it then it just muddies the application. Again the more code there is, the harder it is to make a change.
  • Source control has your back

6. Lots of code is worse than bad code

  • While lots of code is probably just bad code
  • It is easier to change a small ammount of bad code than to change a lot of code

7. Always be refactoring

  • Whenever you touch a file, improve it.
  • Don't be afraid to take some time
  • Tests are your safety net.
  • If there are no useful tests.
  • Write or rewrite then.
  • Bad tests are worse than no tests
  • You've taken the time to understand the code
  • Don't waste that time
  • Capture that knowledge with a test

8. Don't try to fix too much at once

  • Estimating wide reaching refactoring is hard
  • Choose your target wisely
  • It is very easy to bite off more than you can chew.
  • If you find that something is taking far longer than you thought, cut your loses and stop.
  • Don't feel bad
  • Check in what you can
  • Break it up some more.
  • You can get the rest of it next time.

9. Seriously, always be refactoring

  • Don't wait until things are completely awful
  • Don't let code block a new feature

10. When things are bad, explain why

  • This is very hard to do
  • Especially with non engineers
  • Ideally refactoring work would be incremental, continuous
  • It shouldn't be necessary to mention it to anyone
  • It should be something you do as part of your own development process
  • But we live in the real world
  • We're human, we cant always see problems coming
  • Left unchecked code can grow into a huge perhaps insurmountable barrier to continued work
  • It is tough for non engineers to understand why this happens
  • When code gets in the way of a new feature
  • Explain what went wrong, why things are hard to change
  • It can be helpful to explain this in terms of design limitations
  • Explain how the system was designed to do a particular thing and how that is incompatible with the new requirement
  • Explain how it can be avoided in the future

11. Try to give well reasoned estimates

  • It helps to be aware of upcoming work ahead of planning
  • Take a look at the backlog
  • This helps you know ahead of time what needs to change
  • So you're not put on the spot and forced to pull an estimate out of your butt
  • It is really hard
  • It gets easier the more you get to grips with the code base
  • You'll know where the choke points are

12. The more you understand, the more frustrated you'll become


if (!this.widgetType) {
  this.widgetType = GB.models.widgetTypes.get(this.get("widget_type_id"));
}
  • This block of code is a constant source of frustration
  • I haven't worked up the courage to do something about it
  • You'll want to change everything, but you cant

13. Concentrate on the things you can change

  • You need victories, no matter how small
  • It relieves the frustration
  • Feeling that you're making some progress does wonders

14. Isolate, and conquer

  • Create little colonies of better code at the edges of your application
  • Write an adapter that helps your better code communicate
    • with the rest of the application in the way you want it to
  • Expand those colonies up through your application
  • Make them spread like a virus
  • Connect those parts together

15. Don't follow project conventions

  • Bullshit to that
  • Adjust them, augment them, try new things. Keep what sticks.
  • Code is supposed to be expressive
  • Express yourself.
  • Come on and do it
  • But don't ever express yourself with comma first variable declaration
  • I will ruin you

16. Write the code you want to be able to write

  • Design your API
  • Don't think about how you're going to implement it
  • It will bias your design
  • Make it pretty
  • Design it for other people
  • Then write the code that actually makes it work

17. Always code as if you want the person who ends up maintaining your code to be your BFF

  • Be nice
  • It will be ok

18. Go for a walk

  • Stop thinking about code
  • This is impossible

19. Get nerdy about coffee

  • Or tea or something
  • Do something that gets you away from your desk for 5 minutes
  • Do something completely mundane that requires your undivided attention.

20. Don't write code in your own time

  • Or do if you really feel inclined
  • But dont feel obliged to do open source
  • It is ok to work 9-6
  • github is not your CV

21. Don't think about work problems while away from work

  • This is impossible
  • I used to have massive problems switching off from work
  • Then I bought a house
  • Buy a house
  • Or have a kid or something
